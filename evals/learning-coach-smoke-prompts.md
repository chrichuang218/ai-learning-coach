# learning-coach 冒烟验证 Prompt

评估重点不是是否输出完整任务单，而是能否像高级私人教练一样主动备课、选择真实入口、一次只展示一个动作，并根据用户反馈动态推进。

## 1. 仅说“开始学习”

```text
开始学习 TS 吧。
```

验收重点：

- 如果存在学习工作区，先读取使命、背景、最近记录和真实项目。
- 不让用户先选择文件、函数、知识点或课程形式。
- 第一轮只给一个可以立即执行的源码动作或一个问题。
- 不展示完整教学路径、预计卡点和所有掌握标准。
- 不默认创建 lesson、HTML、练习文件或临时实验。

## 2. 跨会话继续

```text
继续昨天的源码。
```

验收重点：

- 从最近未完成动作、learning record 或 lesson brief 恢复。
- 不重新进行背景问卷或从第一章开始。
- 优先回到最近真实项目入口。

## 3. 具体概念追问

```text
这段 await 为什么只是暂停函数，却不会卡住线程？
```

验收重点：

- 先直接回答当前问题。
- 贴住用户代码或运行现象解释。
- 可以使用时间线、状态或 Java 对照，但不展开成完整课程。
- 不强制附带作业、验收模板和学习记录。

## 4. 用户仍然没懂

```text
return new Promise((resolve) => setTimeout(resolve, ms)) 还是没理解。
```

验收重点：

- 判断用户卡在值、时间、回调、类型签名还是线程模型。
- 换一种表示方式，例如命名展开、时间线、状态或最小 Run/Debug。
- 每轮只解决一个误解，不只是把原解释说得更长。

## 5. 用户说“学完了”

```text
学完了，Debug 也掌握了，记录一下。
```

验收重点：

- 先使用当前对话、运行和 Debug 的已有证据。
- 证据充分时不重复考试。
- 记录具体能力、真实纠正、能力边界和下一处入口。
- 不把“用户说掌握了”单独当作证据。

## 6. 真实项目与历史练习冲突

```text
我已经选定 mini-claude 作为 TypeScript 学习项目，开始下一课。
```

验收重点：

- 先读取真实 mini-claude 源码。
- 不因为 learning workspace 中存在历史 exercises 就优先选择练习。
- 路径失效时先小范围定位同名仓库；仍找不到才问路径。
- 不创建新的 exercises 目录；历史实验有复用价值时提炼到 `reference/`，个人证据写入 `learning-records/`。

## 7. 没有正式工作区

```text
我想练英语口语，但不知道从哪里开始。你直接带我练。
```

验收重点：

- 不能因为缺少 MISSION 或记录而停住。
- 用合理假设选择一个低风险真实表达动作。
- 第一轮仍保持简单；只有关键信息会改变方向时才问一个短问题。

## 8. Lesson 文件判断

```text
这次源码辅导明天还要继续，需要记住今天走到哪里。
```

验收重点：

- 可以创建简短 Markdown lesson brief。
- brief 面向教练连续性，记录真实锚点、用户起点、第一个动作、卡点和证据。
- 不生成 HTML、自包含教程或复制大量源码。
- lesson 结束后的掌握结果仍进入 learning record，而不是混在 brief 里。

## 9. 应交给 focus-coach

```text
我同时想学 TS、算法和多 Agent，这个月到底先做哪个？
```

验收重点：

- 判断这是主线取舍，不进入三套教学设计。
- 交给 `focus-coach` 选择主线和周期承诺。

## 10. 用户没有提供背景

```text
开始学习 AI Agent 全栈开发。
```

假设当前对话、工作区和学习记录都没有用户背景。

验收重点：

- 先检查已有上下文，确认背景确实缺失，而不是要求用户重复介绍。
- 如果不同背景会明显改变起点，只主动问一个信息增益最高的短问题，例如最熟悉的语言和做过的真实项目。
- 不同时询问学历、工作年限、目标产品、每天时间、学习风格和技术偏好。
- 用户不知道或不回答时，明确采用临时假设，并用一个低风险真实动作继续判断水平。
- 用户明确陈述的背景与教练推断的能力分开记录；后者不能直接写成已验证能力。
- 现实目标和完成证据写入 `MISSION.md`，稳定职业与经验写入 `LEARNER-BACKGROUND.md`。
- 不能仅因为“AI Agent 全栈”范围较大就交给 `focus-coach`；只有出现高成本方向冲突时才升级。

## 11. 创建长期学习工作区

```text
帮我创建一个 learning-lab，作为长期学习工作区，然后开始学习 AI Agent。
```

验收重点：

- 先检查当前 workspace、本地规则和历史记录中是否已经存在或指定学习工作区，避免创建第二套记录。
- 路径已经明确时直接创建；无法安全判断时只问一个放置位置问题，不让用户选择目录结构或文件模板。
- 新独立工作区只创建有实际内容的 `README.md`、`MISSION.md`，以及已有稳定背景时需要的 `LEARNER-BACKGROUND.md`。
- 不预创建空的 `sources/`、`learning-records/`、`reference/` 或 `lessons/`。
- 不默认执行 `git init`，不复制真实项目源码，也不写死开发者个人绝对路径。
- 创建后简短说明路径和记录内容，随后立即开始第一个学习动作，不输出工作区使用教程。
- 如果用户只进行一次短答疑且未要求持久化，不应强制创建工作区。

## 12. 推荐 GitHub 学习项目

```text
我想学习 AI Agent 全栈，但没有项目。帮我找一个 GitHub 上高 Star 的项目，直接带我学。
```

验收重点：

- 先读取或补齐最低必要的用户背景、目标能力和环境约束，再搜索候选项目。
- 使用当前可验证的 GitHub 数据并检查真实 README、配置、目录、运行方式、维护状态和 License；不能凭记忆编造 Star 或活跃度。
- Star 只能作为发现候选的信号，最终判断还要考虑目标匹配、可读性、规模、测试和本地可运行性。
- 在后台比较候选，默认推荐一个主项目并给出简短的学习适配理由，不输出长排行榜让用户自己选。
- 只有候选代表不同能力方向时，才展示最多两个替代项或询问一个选择问题。
- 项目确定后登记到 `sources/<project>.md`，源码保留在正常开发目录，不复制进 learning workspace。
- 推荐结束后立即给出真实仓库中的第一个学习动作，不继续停留在资源介绍。
- 如果无法联网或验证仓库现状，明确说明边界，不把未经验证的热门度当作事实。

## 13. 更新学习进度与 ETA

```text
今天学完了，更新一下我的目标进度、项目进度和连续学习。我按现在的节奏还要多久？
```

验收重点：

- 先读取 `MISSION.md`、当前 `sources/<project>.md`、近期 `learning-records/` 和已有 `PROGRESS.md`。
- 当前学习有新证据时先创建或更新 learning record，再修改进度仪表盘。
- 目标能力进度和项目学习范围进度分开计算；项目完成不等于读完所有文件。
- 打卡、阅读、lesson 数量和文件数量不能直接提高掌握状态。
- 贡献日历记录当天最高活动强度，不按同日消息数量累加，也不根据模糊历史补签。
- ETA 给剩余有效学习小时区间、每周节奏、周区间和置信度；少于 3 个有效学习日时标记待校准。
- 用户只要求查看而没有新证据时，可以解释现有状态，不伪造一次进度上涨。

## 通用失败信号

- 让用户先选文件、函数、知识点或课程形式。
- 一上来展示目标、入口、任务、验收、复盘的完整任务卡。
- 未读真实项目就创建玩具练习。
- 默认生成 HTML lesson 或长篇静态讲义。
- 具体问题没有直接回答，反而进入战略诊断。
- 用户没懂时重复同一种解释。
- 只因用户说“懂了”就写入掌握记录。
- 背景缺失时直接猜测为事实、发送长问卷，或因用户不回答而拒绝开始。
- 创建学习工作区时要求用户设计目录、生成整套空结构，或因此推迟当前学习。
- 用户没有项目时只给高 Star 链接清单、编造实时数据，或让用户承担最终筛选工作。
- 把一次性诊断实验保存成长期 exercises 体系，或实验结束后不回到真实项目验证。
- 把连续学习天数、阅读时长或完成 lesson 数直接换成掌握百分比。
- 用一个精确日期承诺“学完”，却不说明范围、节奏和置信度。
- 把旧 study plan 当成永久命令，覆盖用户最新偏好。
- 要求存在外部 Wiki 才能完成学习闭环。
