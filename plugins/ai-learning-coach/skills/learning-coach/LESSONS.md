## 正式定义

Lesson 是一次实时、动态、由教练主导的学习会话，不是预生成的教学文件。

用户通过以下过程完成 lesson：

```text
真实项目或真实任务
  + 对话
  + Run / Debug / 实际操作
  + 回答与预测
  + 即时反馈和纠正
  = Lesson
```

用户看到的是眼前一个动作或一个问题。教练在后台掌握完整上下文、可能路径、预计卡点和掌握证据，但根据用户的真实反馈逐步揭示。

## 一个真实 Lesson 如何开始

```text
用户：开始学习 TS 吧。

教练在后台：
读取学习工作区
-> 检查学习者背景和最近证据
-> 打开真实 mini-claude 项目
-> 判断最近发展区
-> 准备第一个问题和可能的验证方式

教练对用户：
请打开 mini-claude 的 src/memory.ts，
搜索 startMemoryPrefetch。
找到后告诉我它的返回类型。
```

接下来由用户的回答决定教学路径。完整过程才是一节 lesson。

## 默认不创建 Lesson 文件

大多数 lesson 不需要对应文件。长期真正有价值的产物是：

- `learning-records/`：用户实际证明掌握了什么、曾经错在哪里、下一处入口是什么。
- `reference/`：以后仍值得复习的稳定知识。
- `NOTES.md` 或学习者背景：怎样教这个用户最有效。
- `STUDY-PLAN.md`：当前短主线和最近几个真实锚点，可选。

不要为了“每节课都要有文件”创建 lesson。文件数量不是学习进度。

## 什么时候创建 Lesson Brief

只有两种默认理由：

1. **跨时间恢复**：一次辅导需要跨天或跨会话继续，现有学习记录不足以恢复现场。
2. **教学设计复用**：这套真实任务、卡点顺序和验证方式以后值得给其他相似学习者复用。

用户要求一份教程或讲义时，可以创建独立文档，但不要自动把它归类为 lesson brief。Lesson brief 的读者首先是教练。

## Lesson Brief 应该很短

它只保存会影响教学判断的内容：

- 为什么现在选择这个入口。
- 真实项目或任务锚点。
- 用户当前已经能做什么。
- 尚未验证的能力。
- 第一个可见动作。
- 预计可能卡在哪里。
- 什么证据足以判定本轮掌握。
- 实际教学结束后走过什么路径。

不要复制完整源码，不要预写整章解释，也不要把所有后续问题提前展示给用户。

## 推荐模板

```markdown
# {Lesson 标题}

状态：准备中 | 进行中 | 已完成 | 已转入其他入口
轨道：{可选}
更新时间：{YYYY-MM-DD，可选}

## 为什么现在做

{根据使命、已有证据和最近卡点，说明为什么选择这里。}

## 真实锚点

- `{真实项目路径或任务}`：`{符号、函数、题目、作品片段}`

## 用户起点

已经证明：

- {...}

尚未验证：

- {...}

## 教练的第一个动作

{只写第一步。不要在首次回复中展示后续计划。}

## 预计卡点

- {...}

## 掌握证据

用户能够：

1. {...}
2. {...}

## 实际教学路径

Lesson 结束后补充：

- 用户首先卡在哪一行或哪一步。
- 提出了什么问题。
- 使用了什么 Run、Debug、作品或真实反馈证据。
- 哪个误解被纠正。
- 哪种解释或教学动作有效。

## 结果

- 学习记录：`learning-records/...` 或“无需创建”
- 稳定知识：`reference/...` 或“尚未稳定”
- 临时实验：未使用 / 已清理 / 结论已沉淀到 `reference/...`
- 下一处真实入口：{...}
```

## 示例：追踪 Memory Prefetch 进入模型请求

```markdown
# 追踪 memory prefetch 进入模型请求

状态：进行中
轨道：typescript-agent

## 为什么现在做

用户已掌握 Promise、async 和 await，但尚未把这些能力映射回 mini-claude 真实源码。

## 真实锚点

- `src/memory.ts`：`startMemoryPrefetch()`
- `src/agent.ts`：`chatOpenAI()`
- `src/agent.ts`：`callOpenAIStream()`

## 用户起点

已经证明：

- async 函数立即返回 pending Promise。
- await 暂停当前函数而非线程。
- 能解释 Promise 的 resolve / reject。

尚未验证：

- 能否在真实 Agent 中独立追踪 Promise 的创建和消费。

## 教练的第一个动作

让用户打开 `src/memory.ts`，搜索 `startMemoryPrefetch`，只回答它的返回类型可能表达什么。

## 预计卡点

- `MemoryPrefetch | null`
- `Promise<RelevantMemory[]>`
- `settled` 与 `consumed`
- sideQuery 的来源
- OpenAI 与 Anthropic 两条对应链路

## 掌握证据

用户能够在不看提示的情况下指出：

1. prefetch 在哪里启动。
2. 为什么启动时不 await。
3. 结果在哪里消费。
4. messages 在哪里修改。
5. messages 在哪里传给 SDK。

## 实际教学路径

课程结束后再补。

## 结果

- 学习记录：课程结束后决定。
- 稳定知识：仅在知识稳定后创建。
- 下一处真实入口：课程结束后决定。
```

## 与其他产物的区别

| 产物 | 回答的问题 |
| --- | --- |
| 实时 lesson | 教练此刻如何根据用户反馈带学？ |
| lesson brief | 教练为什么选这里、准备怎么带、实际怎么走？ |
| learning record | 用户最终证明掌握了什么、误区如何被纠正、下一步是什么？ |
| reference | 哪些知识已经稳定，未来可以独立复习或被知识库摄取？ |
| 临时实验 | 是否曾为隔离机制临时使用，结论最终沉淀到哪里？它本身不是长期产物。 |

## 禁止事项

- 默认生成 HTML lesson。
- 把 lesson brief 写成给用户首次阅读的教材。
- 在用户开始前展示完整教学路径和所有验收问题。
- 复制大量真实源码，形成平行项目。
- 因为创建了 lesson 文件就宣称用户完成学习。
- 每次会话都机械创建一个 lesson 文件。
