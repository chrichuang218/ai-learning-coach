# 学习工作区格式

学习工作区是私人教练的长期记忆，不是课程仓库。它保存使命、用户背景、真实材料入口、掌握证据、学习进度和稳定知识，让下一次会话可以直接继续。

所有文件和目录都采用懒创建。用户只进行一次短学习时，可以完全不创建工作区。

## 发现与创建协议

优先复用现有工作区。包含 `MISSION.md`、`LEARNER-BACKGROUND.md`、学习记录或明确学习导航的仓库，可以视为候选工作区；但普通源码仓库不能仅因正在被学习就自动改造成学习工作区。

创建工作区前按以下顺序确定路径：

1. 用户已经明确指定的路径。
2. 当前仓库、本地规则或历史记录已经登记的学习工作区。
3. 当前 workspace 明确就是新建 learning workspace 的目标位置。
4. 仍无法安全判断时，只询问一次放置位置。

不要在公共 Skill 中写死作者或某位用户的绝对路径。不要让用户决定内部目录结构；这是教练和本格式的职责。

适合创建的时机：

- 用户明确要求创建 `learning-lab` 或长期学习工作区。
- 学习将跨会话继续，且已经有使命、稳定背景或掌握证据需要保存。
- 已有工作区缺少当前确实需要的最小文件。

不适合创建的时机：

- 一次短答疑或临时概念解释。
- 尚无任何可写入内容，只是为了提前生成目录。
- 创建动作会打断当前更重要的真实学习任务。

## 最小初始化

新建独立工作区时，先创建：

```text
learning-workspace/
  README.md
  MISSION.md
  LEARNER-BACKGROUND.md     # 已有稳定背景时创建
  PROGRESS.md               # 用户需要长期感知进度时创建
```

- `README.md` 只说明该实例的用途、核心文件和如何继续，不复制通用 Skill 文档。
- `MISSION.md` 保存当前现实目标和完成证据；未完全确认时明确标记为临时使命。
- `LEARNER-BACKGROUND.md` 只在已有可记录背景时创建，不生成空模板。
- `PROGRESS.md` 只在目标或项目学习范围足够明确且用户需要长期进度时创建。
- 其他目录在出现第一份真实内容时再创建。
- 不默认执行 `git init`，也不复制被学习项目的源码。

如果在已有仓库内使用学习工作区，不覆盖原有 `README.md`，只增加当前确实需要且符合本地规则的学习文件。

## 单主题工作区

优先使用单主题结构：

```text
learning-workspace/
  README.md                 # 独立工作区推荐；已有项目中不覆盖原文件
  MISSION.md
  PROGRESS.md               # 可选用户进度仪表盘
  LEARNER-BACKGROUND.md     # 可选
  NOTES.md                  # 可选
  STUDY-PLAN.md             # 可选短主线
  sources/                  # 可选真实项目/材料登记
  learning-records/         # 有掌握证据时创建
  reference/                # 有稳定知识时创建
  lessons/                  # 可选教练 brief，默认不存在
```

职责：

- `README.md`：当前工作区实例的用途、导航和继续方式，不承载通用创建规则。
- `MISSION.md`：为什么学、学到什么程度、服务什么现实结果。
- `PROGRESS.md`：掌握状态从使命、项目范围和学习记录派生；同时保存紧凑贡献日历、最近活动与 ETA。
- `LEARNER-BACKGROUND.md`：已经证明的能力、已有经验和当前能力边界。
- `NOTES.md`：稳定教学偏好、有效讲法和反复出现的卡点。
- `STUDY-PLAN.md`：当前短主线和最近几个真实锚点，不是完整课程表。
- `sources/`：登记真实项目、题库、作品、官方资料和外部读本入口。
- `learning-records/`：个人化掌握证据和下一入口。
- `reference/`：稳定、去个人化、可复用的 Markdown 知识源。
- `lessons/`：跨时间恢复或复用教学设计时使用的简短教练 brief。

## 多轨道工作区

只有用户确实维护多个长期方向时才升级：

```text
learning-workspace/
  README.md
  MISSION.md
  PROGRESS.md               # 当前主线的用户进度仪表盘
  TRACKS.md
  NOTES.md
  tracks/
    <track>/
      MISSION.md
      LEARNER-BACKGROUND.md  # 可选
      STUDY-PLAN.md          # 可选
      RESOURCES.md           # 可选
      NOTES.md               # 可选
  sources/
  learning-records/
    <track>/
  reference/
    <track>/
  lessons/
    <track>/                 # 可选
```

多轨道只解决长期材料归属，不代表同时推进多个主线。需要选择主线时使用 `focus-coach`。

## 权威顺序

工作区内容发生冲突时，按以下顺序判断：

1. 用户当前和最近明确表达的意图与偏好。
2. 当前仓库的硬性本地规则。
3. 最近学习记录中的证据、纠正和下一步。
4. 当前使命、真实项目登记和仍有效的约束。
5. 较早的 study plan、lesson brief 和历史课程形式。

`PROGRESS.md` 的掌握状态是以上事实的派生视图，最近活动则保存紧凑出勤事实。内容冲突时修正进度文件，不能用旧百分比覆盖新证据。

旧计划是可修订状态，不是永久命令。用户后来明确改为“真实项目 + 对话带学”时，不能继续因为旧文件写着“HTML/练习先行”而沿用旧方式。

## 产物流转

```text
MISSION.md + sources/
目标与项目学习范围
        ↓
真实项目 + 对话 + Run/Debug/实际操作
        ├──────────────→ PROGRESS.md
        │                最近活动与贡献日历
        ↓
learning-records/
个人证据、误区、纠正、下一入口
        ├──────────────→ PROGRESS.md
        │                目标/项目进度与 ETA
        ↓
reference/*.md
稳定、去个人化、可复用的知识源
        ↓ 可选
Obsidian / LLM Wiki / Notion / 其他知识系统
```

Lesson 发生在最上面的实时过程里。Lesson brief 只在需要恢复或复用时旁路保存，不是每次都必须经过的层。

## 临时实验生命周期

临时最小实验不属于学习工作区的长期产物，也不建立标准目录。

1. 优先在真实项目中使用目标测试、Run、Debug、日志或可逆修改隔离问题。
2. 不适合写入真实项目时，使用系统临时目录或明确可丢弃的 scratch 位置。
3. 实验只回答一个问题，完成后立即回到真实项目验证映射。
4. 一次性文件由创建它们的教练清理；不要删除或覆盖用户原有内容。
5. 可复用的机制、命令和预期输出整理到 `reference/`。
6. 用户通过实验表现出的掌握证据写入 `learning-records/`。

历史工作区中的旧练习目录可以保留作为旧记录，但新流程不继续向其中写入。确认没有长期价值后可以归档或删除；有价值的内容先提炼到 `reference/` 或 `learning-records/`。

## 写入规则

- 先判断写入是否会改善下一次教学或长期复习。
- 只有有证据的个人学习结果进入 `learning-records/`。
- 有掌握证据或教学状态变化时先更新 record，再按 `PROGRESS-FORMAT.md` 更新进度；只有低强度出勤时只更新 progress，且不改变掌握百分比。
- 只有稳定、可复用的内容进入 `reference/`。
- 默认不创建 `lessons/`；创建时遵循 `LESSONS.md`。
- 不复制大型真实项目源码，只登记路径并引用相关符号。
- 不要求存在外部 Wiki 才能完成学习闭环。
- 不预创建空目录树，不默认初始化 Git，也不把真实源码复制进工作区。

## 边界

- 工作区不是资料收藏夹。
- `STUDY-PLAN.md` 不是完整课程大纲。
- `reference/` 不是首次学习入口。
- `learning-records/` 不是日记。
- `PROGRESS.md` 不是第二份掌握证据，也不是按课时自动上涨的经验条。
- 临时实验不是长期目录或第三类知识库。
- `lessons/` 不是 HTML 教材目录。
- `assets/` 如存在，只保存真正跨材料复用的展示基础设施，不承载主知识。
