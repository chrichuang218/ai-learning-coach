# Coaching Modes：私人教练训练模式库

训练模式是教练在后台选择的教学动作，不是固定课程流程。每次只选最能暴露当前卡点的一种，用户前台仍然只看到一个动作或一个问题。

## 什么时候读取

- 用户说“还是没懂”，当前解释没有改变心智模型。
- 用户能复述定义，却不能预测、操作或迁移。
- 用户答案正确，但推理过程不可靠。
- 需要从真实项目中隔离一个明确机制。
- 用户已有基础，需要检验真实压力下的能力。

不要为了展示教学技巧而堆叠模式。模式服务当前卡点，不是 lesson 的节目单。

## 快速选择

| 用户状态 | 优先模式 |
| --- | --- |
| 一句话都抓不住 | 钥匙句解锁 |
| 不知道代码何时执行 | 时间线与状态展开 |
| 听懂但不会做 | 错误驱动真实任务 |
| 预测和结果不一致 | 预测－观察－解释 |
| 答案正确但理由含糊 | 思考过程审查 |
| 声称掌握但证据不足 | 掌握度漏洞检测 |
| 会做但讲不清 | 反向教学 |
| 只能在原例子里使用 | 迁移挑战 |
| 基础稳定但实战慌乱 | 高压场景演练 |

## 钥匙句解锁

适合：抽象概念、陌生源码、公式或机制完全没有抓手。

做法：

1. 找一句能打开后续理解的话。
2. 只解释这句话，暂时不展开完整背景。
3. 用当前代码、任务或日常类比落地。
4. 只问一个能暴露当前理解的问题。

示例：

```text
Promise 是未来结果对象，不是线程。

现在只回答：`const p = recallMemory()` 这一刻拿到的是 memory，还是代表未来 memory 的对象？
```

停止条件：用户能用自己的话说清钥匙句，并映射到当前材料。

避免：连续问 2-3 个问题、堆术语、把类比当成精确定义。

## 时间线与状态展开

适合：async、事件、生命周期、工作流、状态机、消息流和跨函数调用链。

做法：

1. 给匿名动作命名。
2. 标记开始、暂停、状态变化、恢复和结束。
3. 一次只追一个值、Promise、消息或状态位。
4. 需要时用日志、断点或图示验证。

统一模板：

```text
启动
-> 同步执行
-> 状态/值产生
-> 当前函数暂停或分支切换
-> 其他工作继续
-> 条件满足
-> 恢复
-> 结果进入下一层
```

停止条件：用户能预测下一步并说明恢复条件。

避免：一开始画完整系统架构图，让局部卡点被更多信息淹没。

## 错误驱动真实任务

适合：用户经常“看懂但不会做”，或只有跟做经验。

做法：

1. 从真实项目、题目、作品或任务切一个小动作。
2. 让用户先做出预测、代码、草稿、解题步骤或判断。
3. 出错后先定位错误模型，不立即倾倒完整答案。
4. 卡死时直接给必要解释，不进行猜谜式折磨。
5. 修正后回到原任务验证。

停止条件：用户能解释错误原因并完成修正后的真实动作。

避免：尚未读取真实材料就制造玩具错误；把犯错本身当成学习目标。

## 预测－观察－解释

适合：代码运行、Debug、实验、数据、考试步骤和可观察行为。

做法：

1. 运行前只要求一个关键预测。
2. 使用 Run、Debug、实验或真实反馈观察结果。
3. 只解释预测与事实之间的差异。
4. 让用户用新模型再次预测相邻情况。

示例：

```text
先别运行。删掉这一行 `await` 后，你预测哪条日志会先出现？
```

停止条件：新预测与事实一致，用户能解释为什么。

避免：先公布答案；一次要求预测所有输出。

## 思考过程审查

适合：答案正确但过程不明，可能靠模板、记忆或碰运气。

检查：

- 每一步为什么这样做。
- 换一个条件后做法是否变化。
- 哪个事实支持当前判断。
- 方法在哪些边界失效。

第一动作通常是：

```text
答案对了。现在只解释你为什么先检查这个条件，而不是另一个条件。
```

停止条件：结果和理由都能站住，并能说明至少一个边界。

避免：把正确答案也变成长时间审讯。

## 掌握度漏洞检测

适合：用户说“已经掌握”，但现有证据只到识别或复述。

做法：

1. 先复用已有证据，不重复考试。
2. 找一个最可能的盲点。
3. 只给一个能区分“会背”和“会用”的问题或动作。
4. 证据充分后立即停止。

证据层级：

```text
识别 < 解释 < 预测 < 独立操作 < 迁移
```

停止条件：达到当前主题所需的最低充分证据。

避免：每次“学完了”都固定出题；为了证明严格而过度考核。

## 反向教学

适合：用户能做但讲不清，或需要准备面试、教学和复盘。

做法：

1. 教练扮演初学者。
2. 用户每使用一个关键术语，就要求给当前例子。
3. 错误时沿逻辑追问一层，让矛盾显现。
4. 最后让用户重写一句更准确的解释。

停止条件：用户能用非循环定义解释机制，并给出例子和边界。

避免：故意装傻太久；沿错误逻辑追问到用户挫败。

## 迁移挑战

适合：用户只会原例子，尚未证明能识别相同机制。

做法：

1. 保留核心机制，只改一个表面条件。
2. 优先使用真实项目中的相邻代码或同类真实任务。
3. 让用户先指出“什么没变、什么变了”。
4. 再完成新的预测或操作。

停止条件：用户能识别同一机制并独立调整。

避免：一次改变太多条件，导致无法判断失败原因。

## 高压场景演练

适合：基础已经稳定，但真实应对速度、取舍或抗干扰能力不足。

可加入一个约束：

- 时间有限。
- 信息不完整。
- 不能查资料。
- 输入包含错误或噪声。
- 需求中途变化。
- 必须明确取舍。

演练后只指出具体失败点：判断慢在哪里、依赖材料在哪里、动作顺序哪里混乱、什么证据没有检查。

停止条件：用户在约束下仍能保持核心动作顺序。

避免：用于完全初学者；同时叠加多个压力源。

## 模式组合规则

同一 lesson 可以先后使用多个模式，但同一轮只展示一个动作。

常见组合：

```text
钥匙句解锁
-> 预测－观察－解释
-> 迁移挑战
```

```text
错误驱动真实任务
-> 思考过程审查
-> learning record
```

```text
掌握度漏洞检测
-> 反向教学
-> reference 稳定知识
```

模式是后台决策。不要把模式名称、完整流程和后续考题提前展示给用户。
