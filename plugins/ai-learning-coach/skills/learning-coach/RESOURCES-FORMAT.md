# RESOURCES.md 格式

`RESOURCES.md` 记录少量高质量资源和真实学习材料。技术工作区也可以使用 `sources/` 为真实项目单独登记路径和阅读重点。资源必须服务当前使命和真实动作。

## 模板

```md
# Resources: {学习主题}

## Primary

- [{资源名称}]({链接或位置})
  {为什么可信，适合解决什么问题，当前该怎么用。}

## Practice Anchors

- {真实任务、项目、题库、作品、场景、论文、数据集、素材或练习入口}

## Communities

- {可获得真实反馈的社区、同伴、老师、论坛或线下场景}

## Parking Lot

- {以后可能有用但当前不追的资源。不要让它们干扰下一步}
```

## 规则

- 资源越少越好。优先一个高质量资源，而不是一串链接。
- 每个资源都要说明服务哪个动作。
- 不把收藏资源当作学习进展。
- 如果资源没有被使用、验证或复盘，不要把它写成核心材料。
- 项目路径失效时更新登记，不要静默切换到玩具练习。
- 外部知识库、社区或课程都不是 learning-coach 开始带学的前置依赖。

## GitHub 项目选择

用户没有真实项目时，可以从 GitHub 或其他代码托管平台推荐项目。先读取用户使命、背景和能力边界，再评估候选：

- **目标匹配**：项目是否包含用户真正想掌握的能力，而不只是标签相似。
- **技术栈匹配**：哪些旧经验可以迁移，哪些新技术会形成合理挑战。
- **可读性与规模**：是否能找到清楚入口，仓库是否大到掩盖当前机制。
- **可运行性**：安装、配置、示例和本地启动是否可复现。
- **维护状态**：近期提交、Issue、Release 和依赖状态是否显示项目仍可使用。
- **教学基础设施**：文档、测试、类型、示例和模块边界是否支持验证理解。
- **许可与风险**：License、外部服务、密钥、成本、平台限制和已知安全风险。

Star 是社区关注度信号，只用于发现候选。推荐结论不能只写“Star 很高”。使用实时数据时记录核验日期；无法核验时不要写具体数字或声称项目仍活跃。

默认推荐一个主项目。只有真实取舍会改变学习方向时，才提供最多两个替代项，并明确各自代价。

## `sources/<project>.md` 模板

```markdown
# {项目名称}

- 类型：{本地 Git 项目 / 远程仓库 / 题库 / 作品}
- URL：{仓库 URL}
- 本机路径：`{绝对路径；尚未克隆时标记待定}`
- 版本：`{branch / tag / commit}`
- 核验日期：{YYYY-MM-DD}
- 技术栈：{主要语言、框架和运行时}
- 当前用途：{服务哪个使命和能力目标}

## 选择理由

- {为什么适合当前学习者，而不只是项目热门}

## 启动方式

- `{安装或运行命令}`

## 推荐入口

- `{path}`：`{symbol / focus}`

## 学习范围

- {本阶段需要理解、操作或修改的模块、链路或任务}
- {明确不要求读完的区域}

## 项目学习完成标准

- {能够独立完成的源码追踪、运行、Debug、修改或迁移行为}

## 已知约束

- {配置、密钥、成本、平台、规模、维护或 License 风险}
```

只登记源码入口、约定学习范围和学习判断，不把大型仓库复制到 `sources/`。`PROGRESS.md` 的项目进度从这里派生；“项目学完”不等于读完所有文件。
