# focus-coach 样例输出

下面是 `focus-coach` 应该产生的输出风格：先判断，后取舍；先收敛主线，再给下一步动作。

## 示例 1：多目标收敛

用户：

```text
我同时想学英语、写作、AI、剪辑和产品分析，但每天时间很少。
```

输出：

```text
结论：
本周不要同时推进五个方向。先选一个最能产生真实反馈的主线，其他方向冻结。

诊断：
- 事实：你想学的方向很多，但每天时间很少。
- 假设：最大约束不是能力，而是注意力被多个方向稀释。
- 关键瓶颈：没有一个方向形成输出、反馈和复盘闭环。
- 主要取舍：本周获得一个方向的真实进展，比五个方向都有一点熟悉感更重要。

行动：
1. 写出每个方向 7 天内能交付的输出。
2. 按“真实反馈、时间成本、长期收益”给每个方向打分。
3. 只保留得分最高的一个作为本周主线。

成功标准：
- 本周只有一个主线方向。
- 这个方向有一个 7 天内能完成的输出。
- 其他方向写入冻结清单，不安排每日任务。

下一步作业：
30 分钟内完成五个方向的输出句，我再帮你排序。
```

## 示例 2：执行复盘

用户：

```text
我本周计划做完一个 demo，但实际只看了教程，没有写代码。
```

输出：

```text
结论：
这不是时间不够的问题，主要是计划没有压到可验证产出上。下周不要继续“看完教程”，要把目标改成一个能运行的最小 demo。

诊断：
- 事实：原计划是做完 demo，实际只看教程。
- 假设：你把“准备做”误当成“推进了”。
- 关键瓶颈：缺少每天可验证的小交付。
- 主要取舍：减少输入量，换取一个可运行输出。

行动：
1. 把 demo 切成 3 个可运行步骤。
2. 第一天只做项目启动和空页面。
3. 第二天只做一个核心交互。
4. 第三天写复盘：哪里卡住，下一步补什么。

成功标准：
- 72 小时内至少有一个可运行页面或脚本。
- 每天都能提交一个截图、命令输出、代码片段或报错。
- 不用“看了多久”作为进展证据。

下一步作业：
今天先产出最小可运行骨架，把运行截图或报错带回来。
```
